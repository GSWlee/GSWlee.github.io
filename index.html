动手写ORM框架心得

发表于 2021-04-23 | 分类于 实践心得

​ 

 在传统的开发过程中，后台管理系统往往都要通数据库进行操作，传统的开发逻辑就是在应用程序需要调用数据库的地方，将SQL语句作为字符串注入到代码中，然后利用第三方库，例如go的“https://github.com/go-sql-driver/mysql”，python的pymysql库等等，这些第三方库调用你注入代码中的SQL语句访问数据库，进行相应的操作返回相应的结果。但是这种模式有以下两个弊端：
  • 由于不同的sql语句的效率不同，开发人员在开发时要花精力与研究如何编写sql语句来加快底层数据库的执行效率
  • 在底层数据库发生变化的时候，例如从mysql变为oracle，相对应的SQL语句可能会发生变化，因此我们在改变数据库的同时也要修改后端代码，增加工作量。

ORM全称Object Relational Mapping，即对象关系映射。该框架就是为了解决这种问题诞生的。他将数据库中的数据映射为对象，开发人员就像调用对象，类一样就能调用数据库，而ORM同时将开发人员的编写的语言转化为底层数据库能处理的语句(SQL等)，将代码与底层数据库解耦，使得开发更加有效率。

发表于 2021-04-13 | 分类于 论文笔记

[2019CLOUD]Exploring Potential for Non-Disruptive Vertical Auto Scaling and Resource Estimation in Kubernetes

​ 本文主要提出了一种对于容器分配资源的垂直扩展方法。在容器运行任务时，通过算法，调整分配的容器的资源，提高整个集群的资源利用率。在本文中，核心的目标是提升资源的利用率，并且容器所部署的虚拟机集群不会发生变化。因此，本文只考虑了容器层面的垂直扩容，至于容器的水平扩容与虚拟机额扩容并不在本文的考虑范围之内。

主要工作：

  • 提出了一种基于k8s的垂直扩展算法，在尽量不干扰容器运行的情况下进行容器的垂直扩展
  • 通过多次实验确定了进行扩展算法的时间周期，平衡了资源利用率与容器扩缩容带来的开销
  • 通过实验验证的算法的有效性

容器云调度论文笔记(二)

发表于 2021-03-31 | 分类于 论文笔记

[2020 TPDS] Elastic Scheduling for Microservice Applications in Clouds


本文主要介绍了一种针对微服务场景下的容器云调度策略。主要是针对多工作流任务在异构集群上进行调度，其目的是在保证工作流在deadline之前完成的情况下尽可能的减少服务器的租凭成本。这篇论文比较精彩的地方就是对于微服务场景下的问题建模，模型很贴合容器云与微服务的场景。本文考虑了在容器云场景下的两成调度问题(task到容器，容器映射到虚拟机),将容器与任务解耦，不是简单的将任务与解决它的容器打包进行调度。

本文的主要工作：

  • 针对容器云在微服务场景下的任务调度进行建模
  • 基于统计信息，提出了一种容器所用的资源配置策略
  • 提出了一种基于任务紧急程度的调度策略
  • 提出了一种容器与虚拟机水平扩展的策略

容器云调度论文笔记(一)

发表于 2021-03-24 | 分类于 论文笔记

[2019 BIG DATA]Progress-based Container Scheduling for Short-lived Applications in a Kubernetes Cluster


本文主要介绍了一种基于集群节点内部资源竞争强度的调度策略，该调度策略减少了减少批处理任务的运行时间。该方法主要针对对于Short-lived Applications(即有着较短生命周期的任务，通常为计算密集型任务，例如map-reduce，神经网络上的训练等等)。在之前的调度算法中，调度器通常只会考虑节点当前的状态，，包括节点的目前的资源利用率，节点上容器的数量等等，本文的算法中，调度起不再只考虑目前的节点上的资源利用情况，也考虑节点未来短期的资源利用情况。

主要工作：

  • 提出了竞争率这个概念，用于衡量单个节点内的资源竞争强度
  • 提出了ProCon,通过ProCon可以监视运行时容器的进度并且预计完成时间。通过这些收集到的数据，计算每个节点的竞争率，给调度器的调度决策做参考
  • 将ProCon部署到K8s集群上，对本算法的性能进行评估
